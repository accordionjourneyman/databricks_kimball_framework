# Type 7, key brokering, identity maps, and null semantics

This is the normative key-management contract for the framework. It describes
implemented behavior. SQL in the final section is explicitly labeled as a
consumer template and is not generated by the framework.

## Why Type 7 has two keys

An SCD Type 7 dimension is a Type 2 history table exposed through two equality
join paths:

- The version surrogate key identifies exactly one historical row. The
  framework derives it from the canonical natural-key payload plus the
  business-effective timestamp.
- The durable key identifies the entity and is identical across every version.
  The framework derives it from only the canonical natural-key payload.

Both are signed `BIGINT` columns. A fact relationship declared as `type7`
receives both keys before the fact is written. The historical query joins the
fact's surrogate key to the dimension without a current-row filter. The current
query joins the fact's durable key and filters the dimension to
`__is_current = true`.

```sql
-- As-was: the customer attributes valid when each order occurred.
SELECT d.city, SUM(f.amount)
FROM gold.fact_sales f
JOIN gold.dim_customer d ON f.customer_sk = d.customer_sk
GROUP BY d.city;

-- As-is: all orders grouped by the customer's current attributes.
SELECT d.city, SUM(f.amount)
FROM gold.fact_sales f
JOIN gold.dim_customer d
  ON f.customer_dk = d.customer_dk
 AND d.__is_current = true
GROUP BY d.city;
```

This follows Kimball's Type 7 definition: both keys are stored in the fact,
with separate Type 2 and current-state views. It is not the same as SCD Type 6,
which overwrites current-value attributes across historical rows.

## Dimension YAML

```yaml
table_name: gold.dim_customer
table_type: dimension
scd_type: 7
keys:
  surrogate_key: customer_sk
  durable_key: customer_dk
  natural_keys: [customer_id]
effective_at: updated_at
track_history_columns: [name, city, segment]
null_policy:
  mode: kimball
  attribute_substitutes:
    postal_code: Missing
sources:
  - name: silver.customers
    alias: c
    cdc_strategy: cdf
    primary_keys: [customer_id]
transformation_sql: |
  SELECT customer_id, name, city, segment, postal_code, updated_at,
         _change_type
  FROM c
```

`effective_at` is mandatory. Intervals are half-open:
`[__valid_from, __valid_to)`. Current rows use the concrete exclusive high
date `9999-12-31 23:59:59.999999`; they do not use `NULL`. Adjacent versions
therefore meet at exactly the same boundary without a microsecond gap or
overlap.

## Fact YAML and the centralized key broker

The transformation projects supplier identities and event time. It does not
calculate dimension hashes or perform a range join. The key broker executes
the lookup before fact validation and merge.

```yaml
table_name: gold.fact_sales
table_type: fact
merge_keys: [order_line_id]
depends_on: [gold.dim_customer]
foreign_keys:
  - column: customer_sk
    references: gold.dim_customer
    dimension_key: customer_sk
    relationship: type7
    durable_column: customer_dk
    durable_dimension_key: customer_dk
    lookup:
      source_columns: [source_customer_id]
      dimension_columns: [customer_id]
      event_time: order_at
      early_arriving: skeleton
      invalid_action: error
sources:
  - name: silver.order_lines
    alias: o
    cdc_strategy: cdf
    primary_keys: [order_line_id]
transformation_sql: |
  SELECT order_line_id, source_customer_id, order_at, amount, _change_type
  FROM o
```

`dimension_columns` defaults to `source_columns`. Declare it when supplier and
dimension names differ. Composite identities are supported by parallel lists.
An `identity_map` currently supports one source identity column.

The broker classifies exceptional states in this precedence order:

| Key | Status | Trigger |
| ---: | --- | --- |
| `-1` | `MISSING` | The supplier identity is `NULL`. |
| `-2` | `NOT_APPLICABLE` | `lookup.not_applicable_when` evaluates true. |
| `-3` | `NOT_YET_AVAILABLE` | A concrete, valid identity has no dimension match. |
| `-4` | `BAD_VALUE` | An identity is blank or the required event time is `NULL`. |

These are literal reserved `BIGINT` values, not hashes. Every dimension is
seeded with all four rows, and generated keys are rejected if they enter this
reserved namespace. Consequently downstream consumers use ordinary equality
joins and never need `COALESCE`, null-safe equality, or a `LEFT JOIN` merely to
retain measures.

### Early-arriving policy

`lookup.early_arriving` has three explicit modes:

- `skeleton` inserts an inferred Type 7 member into the dimension before the
  fact lookup. The dimension loader remains the only key authority. When the
  real row arrives, the SCD engine hydrates the inferred member in place and
  preserves its surrogate key and validity start, so already-loaded facts do
  not become orphans.
- `default` assigns `-3` and writes replay evidence to
  `etl_unresolved_dimension_keys`. A later successful replay closes the row as
  `RESOLVED`.
- `error` blocks the fact before the target is mutated.

`invalid_action` independently controls invalid identities: `error` blocks;
`default` assigns `-4`.

The unresolved registry stores the fact table and relationship, hashes and
JSON for fact grain/source identity, event time, source and dimension Delta
versions, first/last batch IDs, and observation/resolution timestamps. It is
idempotently keyed by fact table, relationship, and fact-grain hash.

## Governed temporal identity maps

`lookup.identity_map` names a supplier/MDM-owned Delta control table with this
minimum schema:

```text
source_identity       NOT NULL
canonical_identity    NOT NULL
valid_from             NOT NULL
valid_to               NOT NULL
```

The framework does not infer matches or choose a survivor. Before joining, it
loads the bounded control table and rejects nulls, non-positive intervals,
overlaps for one source, cycles, chains that have not been flattened to the
final survivor, and tables over the 100,000-row validation budget. Mapping is
event-time aware and uses `[valid_from, valid_to)`.

```yaml
lookup:
  source_columns: [supplier_customer_id]
  dimension_columns: [customer_id]
  event_time: order_at
  identity_map: silver.customer_identity_map
```

This is an identity map/cross-reference, not a Kimball multivalued bridge.
Existing facts are never rewritten when a new identity decision arrives.
Replaying a fact applies the identity decision valid at its event time.

For a merger where reports must roll all old durable keys into today's
survivor, a semantic durable-key bridge is still required. The framework
validates and consumes the identity decisions, but does not currently generate
that BI view. See the SQL-only template below.

## Hash integrity

Spark `xxhash64` supplies compact distributed `BIGINT` keys. The framework also
stores SHA-256 fingerprints of the canonical durable and row payloads. Before
every Type 7 merge, one grouped collision gate compares incoming and existing
fingerprints for both key domains. The write fails if one `BIGINT` represents
multiple payloads or equals a reserved member.

This means collisions are detected, not assumed impossible. The SHA-256 values
are integrity evidence; they are not PII tokenization. Security-sensitive
pseudonymization uses the separate keyed HMAC `tokenize` PII strategy.

## Strict no-NULL dimension policy

`null_policy.mode` defaults to `kimball`:

1. Natural keys and `effective_at` are rejected if null.
2. Descriptive nulls are replaced once in the centralized transformation path.
3. Strings use `Missing`; numeric attributes use zero; booleans use false; and
   dates/timestamps use `1900-01-01`.
4. Complex types require an explicit `attribute_substitutes` value.
5. New strict dimension tables declare every stored column `NOT NULL`.

Supplier contract validation runs before substitution, so an upstream null can
still raise a contract finding instead of being hidden. The `legacy` mode is a
migration escape hatch and is rejected by production manifest publication.

## Enterprise bus matrix

The generated bus matrix distinguishes relationship behavior:

- `X`: one standard relationship.
- `H+C`: Type 7 historical surrogate and current durable paths.
- `+I`: supplier identity is canonicalized through a temporal identity map.

Role-playing names remain grouped under the physical conformed dimension.

## SQL-only templates (not generated framework features)

The following semantic view supports current-survivor consolidation after
entity mergers. It is documentation, not framework-managed DDL:

```sql
CREATE VIEW semantic.customer_durable_bridge AS
SELECT
  source.customer_dk AS source_customer_dk,
  survivor.customer_dk AS consolidated_customer_dk
FROM silver.customer_identity_map m
JOIN gold.dim_customer source
  ON m.source_identity = source.customer_id
 AND source.__is_current = true
JOIN gold.dim_customer survivor
  ON m.canonical_identity = survivor.customer_id
 AND survivor.__is_current = true
WHERE current_timestamp() >= m.valid_from
  AND current_timestamp() < m.valid_to;

SELECT bridge.consolidated_customer_dk, SUM(f.amount)
FROM gold.fact_sales f
JOIN semantic.customer_durable_bridge bridge
  ON f.customer_dk = bridge.source_customer_dk
GROUP BY bridge.consolidated_customer_dk;
```

The two Type 7 equality joins shown at the beginning are supported directly by
framework-produced columns. The semantic merger view above must be owned and
deployed by the consuming data product until first-class semantic-view
materialization is added.
